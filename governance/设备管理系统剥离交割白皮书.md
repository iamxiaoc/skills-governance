# 设备管理系统剥离交割白皮书

## 基于源码仓库核心维度的技术解耦与资产去痕清洗执行方案

> **交割战略方针：** 原地主权平移 / 系统不迁移 / 零物理迁移风险 / 客户原地接管云上管理系统

---

## 一、 高维度剥离切面与战略推导结论

### 1. 源码仓私有生态解耦 (技术主权独立)

* **技术依赖去指纹化：** 清洗业务微服务层深度嵌套的母体公司自研运行时框架、专属安全组件及连接管理类 SDK，重建完全中立、可持续独立演进的源码基线。
* **构建环境声明式移交：** 拔除 `pom.xml` 中对母体效能内网强依赖的黑盒编译插件，确保代码在野外中立环境中可以独立成功编译，重塑可移植的流水线契约。

### 2. 品牌特征与对外暴露清洗 (法律确权合规)

* **知识产权确权红线规避：** 彻底清除用户交互界面（Web端、人机交互、移动端制品）以及数据库初始化脚本中的商标（Logo）、企业简称、敏感代码注释及 Git 历史提交指纹。
* **服务寻址完全中立化：** 清查并更替代码与静态配置中硬编码的母体公司专属域名与对外暴露 URL，全面平移并重构为接管方持有的合法中立域名。

### 3. 云上资产原地无感交割 (业务连续性保障)

* **千万级设备运行连续性：** 物理层面不启动任何微服务搬迁、设备切流或网络对调，规避断连风险。系统保持原地运行，通过底层云大账号账单归属权划转与工商主体变更实现原地安全交割。
* **控制与信任边界重塑：** 撤销系统与母体基础设施的免密隐式信任绑定，通过参数化动态注入，将动态运行参数全面改造为支持环境变量与 Kubernetes Secret 的显式注入模式。

---

## 二、 源码仓库核心维度整改与去痕深度拆解

### 1. 核心应用基础框架与构建工具解耦 (Core Application Framework)

> **主要职责：** 核心架构组 / 基础平台部 / DevOps 平台部
> **整改目标：** 规范微服务底层微内核依赖，消除母体公司内部扩展协议在编译期造成的强绑定限制，重建自闭环制品包装。

* **1.1 业务微服务底层基础微内核（自研定制版 RPC / Spring Boot 内核）与运行脚本**
* **母体专有状态：** 系统重度集成了母体内部私有的 Spring Boot 扩展包及自研 RPC 组件，集成了集团专有的通信协议与名字服务；运行脚本强绑定了母体内网环境。
* **接收方中立方案：** 回退至官方开源标准版本（如原生标准 Spring Boot / 开源通用 gRPC 框架），采用标准的容器启动 Entrypoint 规约与基础运行时镜像。
* **代码级整改动作：** 全局清理私有依赖坐标，全面修正因专有 API 移除引发的编译期语法断层；彻底重写容器运行脚本、健康检查脚本与优雅下线（PreStop）逻辑。


* **1.2 专有 Maven 插件与 CI/CD 构建脚本**
* **母体专有状态：** 工程 `pom.xml` 中集成了强依赖母体效能内网环境方可运行的私有 Maven 插件，用于强制执行内部代码安全扫描与合规检查。
* **接收方中立方案：** 基于中立开源的标准云原生多阶段构建流程（Multi-stage Build），向接收方移交清晰、透明的声明式流水线。
* **代码级整改动作：** 在 POM 配置文件中全量剔除非标准的私有构建插件，重写打包生命周期逻辑，在源码根目录下补齐通用的 `Jenkinsfile` 脚本。



---

### 2. 通用工具、视觉资产与暴露面域名清洗 (Ubiquitous Utility Libraries & Assets)

> **主要职责：** 基础开发团队 / 前端技术部 / 法律事务部
> **整改目标：** 严格对标法律确权与反垄断红线，清洗前后台显式特征，完成技术资产的技术主权交割。

* **2.1 通用底层工具包与国际化资源组件（魔改 Guava / Apache Commons / i18n SDK）**
* **母体专有状态：** 基础代码库深度调用了带有母体企业命名空间前缀的魔改版通用工具 SDK，这些类库中包含了大量针对特定内部场景扩展的定制快捷函数。
* **接收方中立方案：** 全面回归开源社区原生技术栈（官方标准 Apache Commons-Lang3 / 原生 Google Guava 组件）。
* **代码级整改动作：** 全局检索并批量更替项目内的 `import` 包路径；针对缺失的母体特定功能函数，通过在服务本地编写中立工具类予以平替。


* **2.2 品牌视觉、LOGO 物料与对外暴露面域名清洗（Logo / 版权文本 / 暴露面 URL）**
* **母体专有状态：** Web 端、移动端前台交互界面、系统主页页脚及数据库脚本中广泛存在母体公司的 LOGO 资源、品牌简称、版权声明及硬编码的母体对外暴露接口 URL。
* **接收方中立方案：** 视觉物料全面更替为接管方企业标志，确权法律文本与服务接口寻址平移至接管方持有的中立合法域名。
* **代码级整改动作：** 地毯式检索并平替前端静态资源包；统一重写前台用户协议、隐私政策与页脚信息；通过 `git-filter-repo` 批量清洗历史提交日志与代码配置中硬编码的母体域名。



---

### 3. 微服务安全、隐私脱敏与审计组件改造 (Security, Privacy & Audit Aspect)

> **主要职责：** 安全架构组 / 各业务研发团队 / 合规专家组
> **整改目标：** 重塑微服务内部的访问校验拦截机制与加解密算法，撤销母体级大网关与内网安全大盘的强耦合绑定。

* **3.1 微服务专属安全组件与分布式准入拦截器**
* **母体专有状态：** 各业务微服务差异化引入了母体内部安全组件，内部校验切面严重依赖母体集团最外层集中式统一 API 网关的鉴权响应。
* **接收方中立方案：** 基于自建中立认证体系，将多服务的差异化校验代码标准化，安全校验逻辑向后移、下沉至系统内部自治。
* **代码级整改动作：** 重写微服务接口拦截器（Interceptor/Filter）代码，改为本地利用标准公钥私钥对 Token 进行自闭环解析，确保存量千万设备原地无感访问。


* **3.2 数据加密算法、数据隐私脱敏与审计日志组件（Crypto / Privacy / Audit SDK）**
* **母体专有状态：** 强行调用母体机密级专属硬件加密机 SDK；隐私脱敏依赖母体平台资产；审计日志实时上报至母体集中式安全管控大屏。
* **接收方中立方案：** 平移至国际/国家通用标准安全加密算法库（如中立开源的 Bouncy Castle 库）；审计组件实施标准本地化落盘。
* **代码级整改动作：** 重写底层数据加密代码，实现算法中立；彻底切断审计日志外流至大厂大屏的通信链路，全量改写为中立的标准 JSON Stdout 输出。



---

### 4. 中间件基础设施连接管理 (Distributed Infrastructure Connectivity Management)

> **主要职责：** 数据架构组 / 云原生专家组 / SRE 运维组
> **整改目标：** 废除云厂商内网特权免密信任机制，将高可用限流降级控制流全量向后、向下平移至服务网格底座。

* **4.1 中间件连接驱动与寻址管理（数据库连接池 / Redis / Kafka 定制客户端 / Rest Client）**
* **母体专有状态：** 中间件连接 SDK 内部集成了母体内部的名字寻址服务，直接采用母体大账号底座提供的特殊免密信任网络连接。
* **接收方中立方案：** 采用开源通用中立的驱动客户端架构（如标准 HikariCP 连接池、原生 Lettuce 客户端、官方标准 Kafka 客户端组件）。
* **代码级整改动作：** 剥离免密和私有网络寻址逻辑，全面改写数据源及消息客户端初始化代码，转化为可通过环境变量或容器 Secret 动态注入的显式安全密码认证。


* **4.2 系统高可用降级控制（限流组件与稳定性 SDK 剥离）**
* **母体专有状态：** 在微服务源码层硬编码引入了私有限流稳定性组件，其动态策略配置与过载管控强依赖于母体运行大盘平台的黑盒回调。
* **接收方中立方案：** 将高可用限流熔断控制权完全从应用源码中抽出，全量下沉至 **Istio 服务网格（Service Mesh）** 基础设施接管。
* **代码级整改动作：** 从微服务源码中彻底拔除限流组件代码及相关切面，将其高可用过载保护逻辑全量重写为 Helm 发布包随附的标准 Istio 声明式网格流量控制规约（VirtualService / DestinationRule）。
